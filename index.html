Observations

I selected the Ionic Framework for this example for the following reasons

  • I think Ionic has a big future so it is used in StackMates to solve the mobile requirement
  • The Ionic brand is consistent in all they communicate
  • Their presence demonstrates a strong lineup for what an aspiring startup / successful business needs.
  • The above projects have a heavy Javascript and Node bias the static sites are built with Ruby

Given that Ionic Framework provides a good bearer of a solution stack for any business that has desires to leverage the web to drive growth and profits. Note with the addition of ecommerce.

Excluding the demo apps and the forum software, a conservative count shows 7 build systems requiring at least 7 CSS settings files to maintain brand consistency and who knows how much common CSS, HTML and Javascript patterns. This does not factor in the need for marketing campaign landing pages or facebook pages as other likely requirements.

Considering you want to keep the project structure for each sub-domain clearly seperated. If you apply this knowledge to my problem and think about the following questions you will begin to understand why StackMates exists.

The StackMates

With StackMates you can create a static website like this one a desktop web and a mobile app with CORS authentication connected to micro-service backend in around 20 mins plus the time to have a coffee while npm install runs.

For your next project you can just copy the entire content of an existing project, edit some config files, change a few requirements and you have a whole new three stacked solution in about 15 minutes without running npm install. Want to add a new sub domain to a new project that is even easier.

So what are StackMates?

StackMates are made up of the following configuration files that do the job of pulling components (mates) into a stack such as…

  • NPM package.json
  • Rework main.css or SASS main.scss
  • Browserify index.js
  • Gulp client build config
  • Hapi server config
  • Dockerize config

The idea is that you could share these StackMates configuration files that would work inside a common engine that would comprise of code in flux from going app specific to being extracted into independent modules.

A side benefit would be that stacks would be easier to compare and analyse to see what trends were developing.

Part of the Dreamineering + StackMates philosophy is delay choosing software solutions until you really know your use case and that most small projects don’t need project management software as you have all you need to communicate goals and track important information in google drive.
