StackMates

You can create a static website like this one a desktop web and a mobile app with CORS authentication connected to micro-service backend in around 20 mins plus the time to have a coffee while npm install runs.

For your next project you can just copy the entire content of an existing project, edit some config files, change a few requirements and you have a whole new three stacked solution in about 15 minutes without running npm install. Want to add a new sub domain to a new project that is even easier.

So what are StackMates?

StackMates are built up from the following configuration files…

  • NPM package.json
  • Rework main.css or SASS main.scss
  • Browserify index.js
  • Gulp config
  • Hapi config
  • Dockerize config - fig, nscale?

The idea is that you developers could share these StackMates configuration files that would work inside a common build engine that would comprise of code going app specific to being extracted into independent modules.

A side benefit would be that stacks would be easier to compare and analyse to see what trends were developing for the purposes of deciding how to invest efforts in education.

Philosophy

Node.js is the central glue layer

The number one goal is to provide concrete examples for how this rapidly expanding micro-culture of web components can be pulled together compare and learn from different strategies.

This engine is largely upon Gulp and NPM and Browerify and Senecajs the glue between components, but it would be pretty sweet to see a completely different engine and strategy to do the same thing using duo and an alternative build system with graftjs for example.

Getting Started

To get started with StackMates you will need to clone a starter project e.g it starts upfront into the project directory of the build engine. The intent is that only bespoke domain logic should live in the project and that common functionality will be pulled in common modules of the StackMates ecosystem.

The intended benefit is that you can save time hunting around for common solutions and making them work in your build system and instead crack on with tackling the complexity of your domain problem.

HTML & CSS

Handlebars, Rework or Sass
Its your choice, but suit up

Handlebars, Rework or Sass

Its your choice, but suit up

The goal is to marry the template approach of patternlab with the component-based UI approach of Suit.css that is used by twitter and used with a twist at medium

You can decide between SASS or Rework. Rework is my first preference because it is very fast and that is what is used by Suit.css. Howeve SASS is available to supports Ionic and there are many other projects out there to learn from.

The templates use handlebars because this is widely adopted and will allow client code to work with a fullstack solution if the use case justifies the move later.

StackMates Manifest

  • main.css - Rework
  • main.scss - SASS

Intent

  • Informative conventions
  • Rapid navigation
  • Well documented CSS

You should be able to read a fragment of HTML and know where the partial is what the intent of styles rules. IDs are should only be used for scrolling to anchors otherwise use data attributes for javascript. It should be possible to open the relevant CSS or JS file by typing four characters that are made clear by looking at markup.

Structure

CSS

This structure can definitely be improved and I won’t be a solution for everyone. But if you suffer from my problem then maybe? But then again perhaps it is too bespoke?

But I’ve been listening very carefully to Fred George’s talks on Programmer Anarchy and Micro-services and based on my experiences beyond the keyboard I think there are a lot of opportunities out there if this can work.

Reasons

I have enjoyed the opportunity to work in some chaotic work environments to maximise profitability and battle the odds to complete projects on time.

An example was taking arbitrage opportunities to deliver mobile traffic at fixed call costs by routing calls from France to the UK then passing those calls to BT who then passed the calls back to France Telecom to terminate. Because France Telecom and BT had not yet differentiated the routing of mobile calls on international routes my company was charged at the fixed rate. As a result my company could undercut France Telecom in France for terminating French Mobile traffic at a healthy margin.

This process was very inefficient with multiples systems and human processes to get through at the same time trying to plug the gaps in our processes and systems so others were not exposing us. Working to develop software solutions to solve these problems taught me that coding solutions to problems is not that hard, you can always buy a book or search for examples.

The biggest problem is correctly identifying and understanding the real problem what value there is in solving it. Then having the infrastructure to get a solution to market as quickly as possible and the biggest problem here is bureaucracy and misguided beliefs about the value of creating by the book software solutions vs implementing solutions to save the business money.

It is for that reason I am very interested in micro-service because from my experience in chaotic work environments I can imagine what opportunities having this architecture would enable. I feel sure any innovative company of the future will be operating on a micro-service architectured backbone.

Progress

The solution is still very early in development but it already scratches a few itches and ti will only get better as more ideas get added and other become cleaner with better documentation.

The client needs work, but I is OK and can easily be built up on and cleaned up. I really want to understand how to tackle the backend services.
