What is the footprint?

Considering The Operating Model That Is Eating The World and how today’s fastest growing, most profoundly impactful companies are redefining the future operating model of business.

What would a typical digital footprint consist of for companies that desires to join the charge in leading this movement? Taking Ionic Framework as an example, the following is an inventory of their digital footprint.

Observations

  • The blog is not a subdomain but this would be typical
  • The Ionic brand is consistent in all they communicate
  • Multiple build systems and CSS theme configurations
  • Heavy javascript and Node bias but also use Ruby and Rails for their soluton stack
  • There is no ionicframework mobile app but this would be a typical requirement
  • It is unclear whether a desktop solution exists to help run their business

A conservative count of github projects shows 7 build, CSS and JS configs to maintains. If you factor in the need for specific marketing campaign pages or facebook pages this number could easily grow.

Questions

  • How much of this code is common but is copied?
  • How much effort goes into updating build systems for each project?
  • How much effort to align consistent branding?
  • How are best UX Patterns maintained across all properties?
  • Why not use JS solutions for forum or ecommerce?

Personal pain points

I was spending too much time updating build configurations and templates than doing real work. I wanted a common build system for all by projects and wanted to stick with javascript as much possible to enhance productivity but with the flexibility to use other languages where there was an obvious benefit.

Solution

Flexibility Scalability Performance Productivity
Don't paint yourself into a corner

Flexibility Scalability Performance Productivity

Don't paint yourself into a corner

Create multiple client sites or apps from a starter project to build a full suite of properties that maximise code reuse with CORS authentication, connected to backend architecture that can be run as a single app or micro-service architecture. All in around 20 mins plus the time to have a coffee while npm install runs.

StackMates are consist of a collection of configuration files comprised of modules that fit together to create fullstack solutions. The creation of StackMates was directly influenced by watching the following talks and the thought that it would great to have a better way to provide guidance for how tiny modules can be combined to solve larger problems.

  • Client: Tiny Modules - Substack
  • Server: Implementing Microservice Architectures - Fred George

Theory & goals

  • Develop a commonly maintained build engine
  • Don’t put build logic in your app specific code
  • Create a combined spotter network for fullstack
  • Developers would share StackMate bundles
  • Only bespoke domain logic lives in github project
  • Develop a consistent evolution path for new npm modules or services
  • Save time hunting around modules
  • Teach new concepts in context of a larger ecosystem

Philosophy

JavaScript and Node.js provide the central pillars.

Client

There is an old rugby adage that if you don’t win the battle upfront you will probably lose the game. By the same token if you provide an engaging user experience you won’t win customers. The idea is that by starting upfront you don’t lose your story in the backend and create application that customer want to use. This approach is heavily influenced by the thoughts of Ryan Singer in The Art of Designing for People.

Services

The number one goal is to provide concrete examples for how this rapidly expanding micro-culture of web components can be pulled together compare and learn from different strategies.

Alternatives

It would be interesting see a completely different build engine and strategy using something like duo and graftjs for example.

Learning

Fast absorption of knowledge is key to maximise productivity thefore each major concept of the design will have associated Mind Maps and ANKI flash cards to help you grasp and remember concepts and conventions so you can be as productive as possible.

HTML & CSS

Handlebars, Rework or Sass
Its your choice, but suit up

Handlebars, Rework or Sass

Its your choice, but suit up

The goal is to marry the template approach of patternlab with the component-based UI approach of Suit.css that is used by twitter and used with a twist at medium

You can decide between SASS or Rework. Rework is my first preference because it is very fast and that is what is used by Suit.css. Howeve SASS is available to supports Ionic and there are many other projects out there to learn from.

The templates use handlebars because this is widely adopted and will allow client code to work with a fullstack solution if the use case justifies the move later.

StackMates Manifest

  • main.css - Rework
  • main.scss - SASS

Intent

  • Informative conventions
  • Rapid navigation
  • Well documented CSS

You should be able to read a fragment of HTML and know where the partial is what the intent of styles rules. IDs are should only be used for scrolling to anchors otherwise use data attributes for javascript. It should be possible to open the relevant CSS or JS file by typing four characters that are made clear by looking at markup.

Structure

CSS

Each domain has a shared css manifest. Then each sub-domain has a specific css manifest that references the common base.

Handlebars

Partials and helpers are loaded by reference for the common templates directory but screen views are copied from a reference library into the specific project.

The idea is partial should be pretty stable once and if a variation is needed then it can be copied into the domains’s templates directory.

Build

Gulpjs
Fast is best

Gulpjs

Fast is best

Maintaining the build system across projects is the number one reason StackMates exists. I found I was finding that when I jumped from one project to another I spent more time updating the build system than doing the work intended.

StackMates Manifest

gulp config: sets the paths and options for shared build tasks

Intent

Build system best practice are evolving quickly and I don’t want to be constantly project to reflect the latest and greatest. The intent is to cherry pick the best stuff by keeping an eye on projects like the web starter kit and as far proven so, if the build work after an update to a gulp task or dev module in one project it should work for the rest.

Status

Where is this going?
Reality check

Where is this going?

Reality check

The structure can be improved and it won’t be a solution for everyone. But I’ve been listening to Fred George’s talks on Programmer Anarchy and micro-service architecture on repeat and based on my experiences beyond the keyboard I am sure there are a lot of opportunities out there to be taken with this architecture in hand.

Progress

The apps are still largely ignored but it is fairly trivial to convert and angular module to work in this structure and angular modules are becoming increasingly ubiquitous judging from the output of dailyjs. The solution is still very early in development but it already scratches a few itches and will only get better as more ideas get added and other become cleaner with better documentation.
